# PRD: IT Computer Maintenance Module — BCA Platform

## Project Context

The BCA Platform is an internal operations tool for a remote sales team built with:
- Next.js 16 + React 19 + TypeScript (strict)
- Prisma ORM + SQL Server (schemas: dbo, benjaise_sqluser, benjaise_sqluser2)
- NextAuth v5 for authentication and role-based permissions
- TanStack Query v5 for server state
- shadcn/ui (Radix UI) + Tailwind v4 for UI
- CSS custom properties for theming (var(--accent), var(--bg-card), var(--text-primary), etc.)

Agents work remotely and connect to company computers via RemotePC. The IT administrator
periodically connects remotely to perform maintenance: runs CCleaner, Malwarebytes,
cleans temp files, etc. This maintenance must be tracked and scheduled (2-4x per year).
Agents also need the ability to submit maintenance tickets when they have issues
(e.g., "my computer is very slow").

All new code must follow the same patterns as existing modules (calls, clock, sms, clients).
New Prisma models go in schema "benjaise_sqluser2". Navigation and permission keys must be
added following the existing patterns in config/navigation.ts and config/permission-keys.ts.

---

## Goals

1. Provide a centralized registry of all remote computers with owner assignment.
2. Track preventive maintenance cycles with automatic next-due-date calculation.
3. Allow employees to submit corrective maintenance tickets from within the platform.
4. Give the admin a full maintenance history per computer with tools used.
5. Show at-a-glance status: overdue, due soon, up-to-date for each computer.

## Database Schema (Prisma — benjaise_sqluser2 schema)

### Model: Computer
Represents a physical company computer assigned to a remote employee.

Fields:
- id: Int @id @default(autoincrement())
- computerName: String (e.g., "AGENT-PC-01", "BCA-REMOTE-12")
- remotePcId: String? (RemotePC identifier/ID for quick reference)
- assignedUserId: Int? (FK to dbo.Users.IdUser, nullable if unassigned)
- operatingSystem: String? (e.g., "Windows 11 Pro")
- specs: String? NVarChar(Max) (free text, e.g., "i5 8GB RAM 256GB SSD")
- notes: String? NVarChar(Max)
- status: String @default("active") (active | inactive | retired)
- maintenanceIntervalMonths: Int @default(3)
- createdAt: DateTime @default(now())
- updatedAt: DateTime @updatedAt

Relations: user (dbo.Users?), maintenanceLogs[], maintenanceTickets[]
@@map("Computers") @@schema("benjaise_sqluser2")

### Model: MaintenanceLog
A completed maintenance session (preventive or corrective) logged by the admin.

Fields:
- id: Int @id @default(autoincrement())
- computerId: Int (FK to Computer.id)
- maintenanceType: String (preventive | corrective | emergency)
- performedDate: DateTime
- technicianId: Int (FK to dbo.Users.IdUser)
- durationMinutes: Int?
- toolsUsed: String? NVarChar(Max) (JSON array: ["CCleaner","Malwarebytes","TempFiles","WindowsUpdate","DiskCleanup","BrowserCache","StartupOptimization","DriverUpdates","Other"])
- issuesFound: String? NVarChar(Max)
- actionsTaken: String? NVarChar(Max)
- notes: String? NVarChar(Max)
- relatedTicketId: Int? (FK to MaintenanceTicket.id, optional)
- createdAt: DateTime @default(now())

Relations: computer (Computer), technician (User), relatedTicket (MaintenanceTicket?)
@@map("MaintenanceLogs") @@schema("benjaise_sqluser2")

### Model: MaintenanceTicket
An issue reported by an employee requesting maintenance.

Fields:
- id: Int @id @default(autoincrement())
- computerId: Int (FK to Computer.id)
- reportedByUserId: Int (FK to dbo.Users.IdUser)
- subject: String
- description: String NVarChar(Max)
- priority: String @default("normal") (low | normal | high | urgent)
- status: String @default("open") (open | in-progress | resolved | closed)
- resolvedDate: DateTime?
- resolutionNotes: String? NVarChar(Max)
- assignedToUserId: Int? (FK to dbo.Users.IdUser)
- createdAt: DateTime @default(now())
- updatedAt: DateTime @updatedAt

Relations: computer, reportedBy (User), assignedTo (User?), maintenanceLogs[]
@@map("MaintenanceTickets") @@schema("benjaise_sqluser2")

## Computed Logic (Application Layer)

### Next Maintenance Due Date
  lastMaintenanceDate = MAX(performedDate WHERE maintenanceType = 'preventive')
  nextDueDate = lastMaintenanceDate + maintenanceIntervalMonths
  If no logs: nextDueDate = Computer.createdAt + maintenanceIntervalMonths

### Maintenance Status Labels
- "overdue"  → nextDueDate < today
- "due-soon" → nextDueDate within 30 days from today
- "ok"       → nextDueDate more than 30 days away
- "never"    → no preventive maintenance ever performed

## API Routes

All follow Next.js App Router pattern: export async function GET/POST/PATCH/DELETE.
Use Prisma from '@/lib/prisma', auth via getServerSession from '@/lib/auth'.
Return NextResponse.json(). Validate with zod.

### Employee Routes

GET /api/maintenance/my-computer
  Returns Computer assigned to current user.
  Includes: lastPreventiveDate, nextDueDate (computed), maintenanceStatus, openTicketCount.

GET /api/maintenance/tickets
  Returns tickets submitted by current user.
  Query params: status (optional filter).

POST /api/maintenance/tickets
  Body: { computerId, subject, description, priority }
  Creates MaintenanceTicket with reportedByUserId = session.user.id, status = 'open'.
  Validates: computerId must be assigned to current user (admin can override).

### Admin Routes

GET /api/admin/maintenance/computers
  Returns all computers with assignedUser name, lastPreventiveDate, nextDueDate,
  maintenanceStatus, openTicketCount.
  Query params: status, maintenanceStatus, search.

POST /api/admin/maintenance/computers
  Body: { computerName, remotePcId, assignedUserId, operatingSystem, specs, notes, maintenanceIntervalMonths }

PATCH /api/admin/maintenance/computers/[id]
  Body: partial Computer fields

DELETE /api/admin/maintenance/computers/[id]
  Soft delete: sets status = 'retired'

GET /api/admin/maintenance/logs
  Returns all logs with computer name, technician name.
  Query params: computerId, type, dateFrom, dateTo, page, pageSize.

POST /api/admin/maintenance/logs
  Body: { computerId, maintenanceType, performedDate, durationMinutes, toolsUsed (string[]),
          issuesFound, actionsTaken, notes, relatedTicketId }
  If relatedTicketId provided: update that ticket to status = 'resolved', set resolvedDate.

GET /api/admin/maintenance/tickets
  Returns all tickets. Query params: status, priority, computerId, page, pageSize.

PATCH /api/admin/maintenance/tickets/[id]
  Body: { status, assignedToUserId, resolutionNotes, resolvedDate }

GET /api/admin/maintenance/dashboard
  Returns: totalComputers, overdueCount, dueSoonCount, openTicketsCount,
           urgentTicketsCount, recentLogs (last 5), upcomingMaintenance (next 5 due).

## UI/UX Guidelines

CSS variables (always use, never hardcode colors):
  var(--bg-card), var(--accent), var(--success), var(--danger), var(--border)
  var(--text-primary), var(--text-secondary), var(--text-muted), var(--bg-elevated)

Maintenance status → color:
  overdue  → var(--danger)    (red)
  due-soon → amber (#f59e0b)
  ok       → var(--success)   (green)
  never    → var(--text-muted)(gray)

Priority → color:
  urgent → var(--danger), high → orange, normal → var(--accent), low → var(--text-muted)

Use existing components: Badge, Card, CardHeader, CardTitle, CardContent,
Dialog, Select, Input, Button, Loading, Tabs from '@/components/ui/*'.
Toast pattern follows calls page (local state, auto-dismiss 4s).

## Acceptance Criteria

- Admin can register a computer with name, RemotePC ID, and assigned employee.
- System computes correct maintenance status (overdue/due-soon/ok/never) per computer.
- Admin can log a preventive maintenance session with tools used; next-due date updates.
- Employee can view their assigned computer status on the /maintenance page.
- Employee can submit a maintenance ticket (subject, description, priority).
- Admin can view, assign, and resolve tickets, optionally linking a maintenance log.
- All pages are protected by role-based permissions (maintenance / admin_maintenance).
- All forms validate with zod on client and server side.
- UI is consistent with existing platform (dark theme, CSS variables, shadcn/ui).
- No TypeScript errors (strict mode). No 'any' types. Schema-first with zod validation.
